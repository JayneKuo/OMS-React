# OMS 会话式创建商品并按确认发布到渠道

## Quick Start

这个 skill 的入口不是后台接口，也不是脚本命令，而是 OMS 页面里的 AI 输入框。

典型用户输入：

- 帮我创建一个商品
- 帮我把这个 Shopify 商品导入 OMS
- 把这个商品先建到 OMS，成功后同步到 Shopify
- 帮我创建商品，确认后再发到 TikTok Shop

标准执行思路：

1. 先识别用户意图。
2. 先补齐 OMS 创建所需字段。
3. 先在 OMS 建档。
4. OMS 创建成功后，再询问用户是否继续同步到目标渠道。
5. 用户确认后，再执行渠道发布。
6. 输出 OMS 创建结果和渠道发布结果。

默认原则：

- 先创建 OMS，再发渠道
- 优先自动补齐字段，再执行创建
- 创建成功后再做二次确认是否同步到渠道
- OMS 未返回可用 id 时，不要把流程描述成完全成功
- OMS 创建失败时，不继续发 Shopify、TikTok Shop 或其他渠道
- 不到万不得已，不把用户当成表单录入员
- 优先从当前登录用户上下文和 DI 集成配置中自动获取渠道信息、商户信息和授权信息

## Workflow

### 1. 识别用户到底要做什么

先判断用户意图属于哪一类：

- 只创建到 OMS
- 创建到 OMS 后同步到 Shopify
- 创建到 OMS 后同步到 TikTok Shop
- 创建到 OMS 后同步到多个渠道

如果用户没有明确说“自动同步”，默认只做：

- 先建 OMS
- 建完后询问是否继续同步

### 2. 从自然语言中抽取输入

优先识别这些字段：

- 来源渠道：Shopify / TikTok Shop / 其他
- 来源商品标识：商品 id、handle、SKU、链接
- `merchantNo`
- OMS `channelNo`
- 目标状态：草稿 / 上架 / 待审核
- 是否自动发布到外部渠道
- 目标渠道：
  - Shopify
  - TikTok Shop
  - 其他
- 是否需要兼容当前 OMS 页面展示逻辑

并且优先自动推导或自动获取这些字段，而不是先问用户：

- `merchantNo`
  优先从当前登录用户上下文、当前租户、当前商户工作区中获取
- `channelNo`
  优先调用 DI 接口检查当前租户下是否已绑定 Shopify / TikTok 等渠道，如果只有一个可用渠道则自动选中
- Shopify 店铺
  优先调用 DI 接口读取当前租户下已连接的 Shopify 店铺
- 渠道 Access Token
  优先调用 DI 接口读取当前租户下已连接店铺的授权信息
- 价格
  优先自动估算，不默认追问用户
- 图片
  优先使用来源商品图片；如果没有来源图片，再考虑自动生成图片策略

如果用户没有明确说明，按以下默认值处理：

- 来源渠道：`shopify`
- 目标状态：`draft`
- 发布策略：`先保存 OMS，成功后再询问是否同步到目标渠道`
- 映射模式：`compat`

### 3. 把字段分成两批

#### A. OMS 创建必填字段

这些字段必须先补齐，才能在 OMS 建档：

- 商品标题
- 商品描述
- 类目
- `merchantNo`
- `channelNo`
- 商品类型
- 销售属性
- SKU 信息
- 价格
- 库存

#### B. 渠道同步字段

这些字段可以在 OMS 创建成功后再继续确认：

- Shopify 渠道映射字段
- TikTok Shop 渠道映射字段
- 渠道类目
- 渠道属性
- 渠道图片规则
- 渠道发布状态

### 4. 当字段缺失时，先引导用户

不要直接报一句“参数不完整”，也不要一上来把缺失字段全扔给用户。

优先自动完成以下工作：

- 根据商品标题、描述、来源商品信息自动推断类目
- 根据已有 SKU 和来源价格自动给出建议售价
- 根据来源图片自动挑主图、变体图
- 根据商品标题和颜色、尺寸等信息自动生成：
  - `spuName`
  - `sellerParentSku`
  - `sellerSku`
  - `handle`
- 根据来源字段自动整理：
  - 标签
  - 关键词
  - 纯文本描述
- 如果用户只给了模糊描述，如“红色苹果摆件”，优先自动生成一版可创建草稿数据
- 如果当前用户已经登录 OMS，优先从当前用户上下文中读取：
  - `merchantNo`
  - 当前租户
  - 当前工作区商户
- 如果当前系统已接入 DI，优先调用 DI 接口自动获取：
  - Shopify 是否已连接
  - TikTok Shop 是否已连接
  - `channelNo`
  - 店铺标识
  - Access Token
  - 渠道连接状态

只有当这些字段属于高风险且无法可靠推断时，才追问用户：

- `merchantNo` 无法从当前用户上下文拿到
- `channelNo` 无法从 DI 集成自动确定
- 类目无法确定
- 价格无法合理推断
- 库存完全未知且业务不允许默认值
- 变体结构不明确
- 用户没有说明是否允许自动同步渠道

应该像产品助理一样补问：

- 还缺类目时：提示用户补类目
- 还缺价格时：提示用户补价格
- 还缺库存时：提示用户补库存
- 变体太多时：提示用户是否先保留前 N 个变体验证

优先一轮只问最关键的缺失字段，不要一次抛给用户十几个问题。

### 5. 获取源商品数据

如果来源是 Shopify：

1. 调 Shopify Admin API 获取商品详情
2. 读取：
   - 商品基础信息
   - options
   - variants
   - images
   - category
   - tags
3. 归一化成内部中间模型

如果来源是其他渠道，也先完成相同的归一化步骤。

如果来源不是现成商品，而是用户自然语言描述的“新商品”，则自动生成一版最小可创建商品模型，至少补出：

- 标题
- 英文标题
- 默认描述
- 默认类目候选
- 默认 SKU 结构
- 默认价格策略
- 默认库存策略
- 默认图片需求

如果系统中已经有渠道集成配置，优先走：

1. 当前用户上下文
2. 当前租户 / 商户上下文
3. DI 集成接口
4. 来源商品接口
5. 最后才向用户追问

### 6. 选择映射模式

支持两种模式：

#### compat 模式

当 OMS 当前页面展示、SKU 变体渲染、下游发布流程仍依赖已有类目 ID、属性 ID、属性值 ID 时使用。

目标：

- 让 OMS 当前页面先正常展示
- 让 SKU / 变体列表先正常渲染
- 让当前链路先跑通

#### native 模式

当 OMS 已支持按 Shopify、TikTok 等渠道原生字符串识别类目和属性时使用。

目标：

- 不依赖旧渠道 ID 体系
- 直接按 `attributeName + attributeValue` 识别
- 支持 Shopify/TikTok 自己的类目和属性模型

详细规则见：

- [references/mapping-modes.md](./references/mapping-modes.md)

### 7. 构造 OMS 数据

始终先构造：

- `channelProductInfo`
- `omsProductInfo`

#### channelProductInfo

用于保留来源渠道的原始语义，方便：

- 回查
- 重试
- 对账
- 二次发布

#### omsProductInfo

用于保存 OMS 标准结构，至少包括：

- `spuInfo`
- `salesAttributes`
- `productAttributes`
- `skuInfoList`
- `sizeAttributeList`

### 8. 创建前校验

在调用 OMS 创建前，必须校验：

- `merchantNo`
- `channelNo`
- 商品标题
- `itemType`
- `productType`
- `sellerParentSku`
- SKU 列表不为空
- 每个 SKU 有：
  - `sellerSku`
  - `salesPrice`
  - `inventory`
  - 销售属性值

### 9. 大商品先裁剪

如果商品太大，先裁剪后创建 OMS，避免接口异常。

建议裁剪规则：

- `title` 裁到 180~200 字符
- `descriptionHtml` 限长或缩短
- `description` 去 HTML 后裁到 300~500 字符
- `tags` 裁到 180~200 字符
- `images` 先保留主图和必要变体图
- `variants` 首次验证时先保留前 N 个

### 9.1 自动定价策略

如果用户没有明确给价格，优先自动给出建议价格，而不是先追问。

建议优先级：

1. 使用来源商品已有售价
2. 使用来源商品的 `compare_at_price`、成本价、供货价做推算
3. 调用竞品或历史数据接口，比较同类商品价格带
4. 结合类目、规格、库存压力、渠道特征给出建议售价
5. 给出建议价并记录“这是自动估价”

如果存在多个变体：

- 先按规格差异自动拉开梯度价格
- 同时把价格来源标记为“自动估算”

### 9.2 自动图片策略

如果用户没有单独给图片，但来源渠道已有图片：

- 自动选第一张主图作为商品主图
- 自动按变体图片关系绑定 SKU 图
- 自动裁剪过大的图片集合，只保留必要图

如果用户是纯自然语言新建商品，没有现成图片：

- 先创建 OMS 草稿商品
- 明确标记“缺少图片素材”
- 如果系统支持 AI 出图或模板图生成，可继续自动生成：
  - 主图
  - 场景图
  - 变体图

### 10. 先创建到 OMS

调用 OMS 创建接口时：

1. 发 `channelProductInfo`
2. 发 `omsProductInfo`
3. 按用户要求设置：
   - 草稿
   - 已上架
   - 待审核

创建完成后分两种情况：

#### OMS 创建失败

- 直接告诉用户创建失败
- 输出失败原因
- 不继续同步渠道

#### OMS 创建成功

- 明确告诉用户 OMS 已创建成功
- 返回目前已知结果
- 如果没有拿到真实 `productId`，也要明确说明
- 然后继续进入“是否同步到渠道”的确认阶段

### 11. 创建成功后，二次确认是否同步到渠道

标准动作：

1. 告诉用户 OMS 创建成功
2. 再问：
   - 是否继续同步到 Shopify
   - 是否继续同步到 TikTok Shop
   - 是否继续同步到其他渠道

只有在用户确认后，才继续发渠道。

如果用户一开始已经明确说：

- 创建成功后自动同步到 Shopify
- 创建成功后自动发布到 TikTok Shop

才允许跳过二次确认。

### 12. 向目标渠道发布

支持目标渠道：

- Shopify
- TikTok Shop
- 其他后续可扩展渠道

发布步骤：

1. 判断目标渠道
2. 选择对应发布器
3. 生成目标渠道 payload
4. 调目标渠道发布接口
5. 返回发布结果

如果目标渠道是 Shopify：

- 必须使用 UTF-8 编码发送 JSON
- 中文标题、中文描述、中文属性值不能直接按 PowerShell 默认字符串 body 发送
- 优先使用 `scripts/publish-shopify-product.ps1`

如果用户要求“只入 OMS，不上架”，则到 OMS 成功为止。

### 13. 输出结果

输出必须清楚区分三层结果：

1. 商品字段整理 / 补齐结果
2. OMS 创建结果
3. 渠道同步结果

推荐输出节奏：

#### 第一阶段输出

- 已识别商品信息
- 缺失字段
- 是否准备开始创建 OMS

#### 第二阶段输出

- OMS 创建成功/失败
- OMS 返回 id / 未返回 id
- 是否继续同步到 Shopify / TikTok

#### 第三阶段输出

- 渠道同步成功/失败
- 渠道返回结果
- 后续建议

## Decision Rules

- 当用户只是说“帮我创建一个商品”，优先理解成“先创建到 OMS”
- 当 OMS 创建完成后，优先追问“是否继续同步到 Shopify/TikTok”
- 当用户一开始明确要求“创建后自动同步”，才允许省略二次确认
- 当 OMS 当前页面渲染依赖兼容属性 ID 时，优先使用 `compat`
- 当 OMS 已支持原生字符串属性时，优先使用 `native`
- 当创建接口 `success=true` 但没有返回 `productId` 时，不把它当成完全成功
- 当后续发布依赖 OMS 返回 id，而 OMS 没返回时，必须停止并说明原因

## Resource Usage

- 需要查看完整执行链时，读取 [references/workflow.md](./references/workflow.md)
- 需要选择 compat/native 模式时，读取 [references/mapping-modes.md](./references/mapping-modes.md)
- 使用 `scripts/run-workflow.ps1` 模拟 OMS 输入框里的会话式执行流程
- 使用 `scripts/publish-shopify-product.ps1` 发送 Shopify 商品创建或更新请求，并强制使用 UTF-8
- 当这套流程开始反复执行时，优先扩展 `scripts/`，不要把执行细节继续堆进 `SKILL.md`

## Constraints

- 尽可能使用中文输出
- 尽可能采用引导式对话，而不是一次性抛出所有缺失字段
- 尽可能自动补齐字段，只有高风险缺口才追问用户
- 尽可能自动给出建议价格和图片策略，不要默认让用户手动填所有字段
- 先完成 OMS 建档，再决定是否继续同步到外部渠道
- 如果当前兼容方案依赖旧 ID 体系，必须明确说明这是临时兼容方案，不是长期目标方案
- 对 Shopify 的中文字段，必须按 UTF-8 发送，避免标题和描述变成 `????`
- 如果要实现真正的自动商户、自动渠道、自动 token 获取，必须接入当前 OMS 用户上下文和 DI 集成接口，而不是让用户手输这些字段
