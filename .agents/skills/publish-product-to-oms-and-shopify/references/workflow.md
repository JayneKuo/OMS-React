# 工作流细化

## 目标

把“用户在 OMS 输入框里输入一句自然语言”变成：

1. 结构化商品创建请求
2. OMS 商品建档
3. 用户确认后的渠道发布

## 标准执行顺序

1. 识别用户意图
2. 优先从当前用户上下文和 DI 集成里自动获取商户和渠道信息
3. 抽取剩余关键输入
4. 获取源商品数据
4. 归一化商品数据
5. 选择映射模式
6. 判断哪些字段属于 OMS 创建必填
7. 自动补齐能推断的字段
8. 引导用户补齐剩余高风险缺失字段
8. 构造 OMS payload
9. 校验字段
10. 裁剪高风险大字段
11. 创建 OMS 商品
12. 验证 OMS 返回
13. 创建成功后询问是否继续同步到目标渠道
14. 用户确认后执行渠道发布
15. 输出结果

## 会话式引导原则

### 自动获取优先级

在向用户追问之前，先按下面顺序自动获取信息：

1. 当前 OMS 登录用户上下文
2. 当前租户 / 当前商户工作区
3. DI 集成接口
4. 来源商品接口
5. 默认策略推导
6. 最后才问用户

优先自动获取：

- `merchantNo`
- `channelNo`
- Shopify 店铺
- TikTok 店铺
- Access Token
- 已连接渠道状态

如果 DI 中只有一个可用店铺或一个可用渠道，默认直接使用，不先打扰用户。

### 用户输入模糊时

例如：

- 帮我创建一个商品
- 帮我发布这个商品
- 帮我同步一下

优先判断：

1. 来源商品是什么
2. 用户到底是只要建 OMS，还是还要继续同步到渠道

### 字段缺失时

不要直接说“参数不完整”。

优先引导：

- 缺类目就问类目
- 缺价格就问价格
- 缺库存就问库存
- 变体太多就问是否先保留前 N 个

但在追问之前，先尝试自动完成：

- 自动定价
- 自动类目推荐
- 自动规格归一化
- 自动图片选择
- 自动草稿建模

### OMS 创建成功后

进入二段式流程：

1. 明确告诉用户：OMS 已创建成功
2. 再问：
   - 是否继续同步到 Shopify
   - 是否继续同步到 TikTok Shop

### 用户一开始明确要求自动同步时

例如：

- 帮我创建并发布到 Shopify
- 建档后直接同步到 TikTok

可以在 OMS 创建成功后自动继续，但仍要在输出中明确说明：

- OMS 已成功
- 现在开始同步目标渠道

## 失败处理

### OMS 创建失败

如果出现：

- HTTP 失败
- 业务 code 非成功
- `success=true` 但无 id

则：

- 明确告诉用户失败在哪一步
- 输出关键返回信息
- 如果是 OMS 接口本身未返回 id，说明这是接口能力限制
- 不继续执行渠道同步

### 渠道发布失败

如果 OMS 已成功但渠道发布失败：

- 保留 OMS 成功结论
- 单独标记渠道失败
- 提示可重试发布

### Shopify 中文乱码

如果 Shopify 创建后出现：

- 标题变成 `????`
- 描述变成 `????`
- 中文属性值变成 `????`

优先判断为编码问题，而不是业务字段问题。

处理方式：

1. 使用 UTF-8 编码重新发送 Shopify 请求
2. `Content-Type` 使用 `application/json; charset=utf-8`
3. 对于 PowerShell，请以 UTF-8 bytes 发送 body，不要直接发送默认字符串

## 输出建议

最终输出建议带上：

- 来源商品标识
- 映射模式
- 是否裁剪
- OMS 创建结果
- OMS id / 渠道商品 id
- 渠道发布结果
- 风险与后续建议
