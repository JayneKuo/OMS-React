# 调拨单状态与列表异常视图设计

## 1. 背景

当前调拨单模块已经具备主单列表、详情页、创建弹窗和关联单据展示能力，但“调拨主状态”和“下游执行单据创建结果”的表达仍然混杂，导致以下问题：

1. 用户无法快速判断一张调拨单是“主单未建立”、还是“主单已建立但下游建单失败”。
2. 主状态过早进入执行语义，会让用户误解为仓库已开始履约。
3. 列表页缺少面向异常处理的聚合入口，用户难以及时定位失败、部分成功、卡住的单据。

因此，需要补充一层更业务化的“执行单据创建状态”，并在列表页增加“异常”Tab，帮助用户快速处理问题单。

## 2. 目标

本次设计目标如下：

1. 明确调拨单创建后的状态分层。
2. 新增主状态 `PROCESSING`，用于表示下游执行链路已建立完成、进入正式执行阶段。
3. 用“执行单据创建状态”替代技术化的 allocate 文案。
4. 在列表页新增“异常”Tab，聚合所有需要人工关注的调拨单。
5. 尽量复用现有页面结构，避免大范围重构。

## 3. 推荐方案

### 方案 A（推荐）：主状态 + 执行单据创建状态双层表达

- 主状态表达业务执行阶段。
- 执行单据创建状态表达下游建单结果。
- 列表页和详情页同时展示两层状态。

优点：
- 业务语义清楚。
- 能准确表达“主单已创建但下游失败”的场景。
- 便于后续扩展部分成功、重试、异常处理。

缺点：
- 页面信息会增加一列或一块卡片。

### 方案 B：仅扩展主状态

- 把建单失败、部分成功、执行中都塞进主状态。

优点：
- 页面字段更少。

缺点：
- 主状态语义混乱。
- 用户难以区分“业务阶段”和“系统集成阶段”。

### 方案 C：仅保留技术侧 pushStatus

- 不调整主状态，只扩展底层推送状态展示。

优点：
- 实现成本最低。

缺点：
- 页面文案技术感强。
- 产品和业务用户理解成本高。

### 结论

采用方案 A。

## 4. 状态设计

### 4.1 调拨单主状态

建议主状态定义为：

- `DRAFT`：草稿
- `CREATED`：已创建
- `PROCESSING`：执行中
- `IN_TRANSIT`：在途
- `RECEIVED`：已收货
- `CLOSED`：已关闭
- `VOIDED`：已作废
- `CANCELLED`：已取消

#### 状态定义

**已创建（CREATED）**
调拨主单已生成成功，但下游执行单据尚未全部创建成功，调拨尚未进入正式执行阶段。

**执行中（PROCESSING）**
本次调拨所需的下游执行单据已全部创建成功并被下游接受，调拨进入正式执行阶段，等待仓库按单执行。

**在途（IN_TRANSIT）**
真实出库已发生，货物在运输途中。

**已收货（RECEIVED）**
目标仓已收货。

### 4.2 执行单据创建状态

页面展示名称统一为：**执行单据创建状态**。

建议枚举如下：

- `PENDING`：待创建
- `CREATING`：创建中
- `FAILED`：创建失败
- `PARTIAL_SUCCESS`：部分成功
- `SUCCESS`：全部成功
- `CANCELLED`：已取消

#### 规则

1. 调拨单创建成功后，主状态进入 `CREATED`，执行单据创建状态初始化为 `PENDING`。
2. 系统开始异步创建下游执行单据后，执行单据创建状态进入 `CREATING`。
3. 若全部必需单据创建成功且下游接受成功，执行单据创建状态更新为 `SUCCESS`，主状态从 `CREATED` 推进到 `PROCESSING`。
4. 若全部失败，主状态保持 `CREATED`，执行单据创建状态更新为 `FAILED`。
5. 若部分成功，主状态保持 `CREATED`，执行单据创建状态更新为 `PARTIAL_SUCCESS`。

## 5. 状态流转

### 5.1 主状态流转

```text
DRAFT
→ CREATED
→ PROCESSING
→ IN_TRANSIT
→ RECEIVED
→ CLOSED
```

异常流转：

```text
DRAFT / CREATED / PROCESSING
→ VOIDED 或 CANCELLED
```

### 5.2 执行单据创建状态流转

```text
PENDING
→ CREATING
→ SUCCESS
```

异常分支：

```text
PENDING / CREATING
→ FAILED
```

```text
CREATING
→ PARTIAL_SUCCESS
→ CREATING / SUCCESS
```

## 6. 页面设计

### 6.1 列表页

#### 新增字段

在现有列表页中新增一列：**执行单据创建状态**。

展示目的：
- 让用户区分“主单已创建”与“下游执行单据创建是否完成”。
- 支持异常单据的快速筛选与处理。

#### 建议展示文案

- 待创建
- 创建中
- 创建失败
- 部分成功
- 全部成功
- 已取消

#### Tab 设计

现有 Tab：
- 全部
- 草稿
- 在途
- 已完成

新增 Tab：
- **异常**

### 6.2 异常 Tab 定义

异常 Tab 用于聚合需要人工关注和处理的调拨单。

#### 纳入条件

任一满足即可进入异常 Tab：

1. 执行单据创建状态为 `FAILED`
2. 执行单据创建状态为 `PARTIAL_SUCCESS`
3. 执行单据创建状态为 `CREATING` 且超出预期时长（当前前端先不做超时判断，只预留口径）
4. 关联单据存在失败状态（当前前端如暂无数据，可先按执行单据创建状态聚合）

#### 页面价值

异常 Tab 不是主状态分类，而是问题视图。其作用是让用户快速定位：
- 哪些调拨单下游建单失败
- 哪些调拨单只建成了一部分单据
- 哪些调拨单需要重试或人工处理

### 6.3 详情页

详情页新增或强化一块独立区域：**执行单据创建状态**。

建议展示内容：
- 当前状态
- 失败原因
- 已成功创建单据数 / 目标单据数
- 最近创建时间
- 重试次数

该区域应独立于“调拨主状态”展示，避免用户误解。

## 7. 操作设计

### 7.1 在 `CREATED` 状态下

根据执行单据创建状态，页面应支持以下操作：

- `PENDING`：等待自动创建，必要时支持手动触发
- `CREATING`：查看进度，必要时取消
- `FAILED`：查看失败原因，重试创建，取消，作废
- `PARTIAL_SUCCESS`：查看已成功与失败单据，补建缺失单据，重试失败单据
- `SUCCESS`：自动或手动推进主状态为 `PROCESSING`

### 7.2 在 `PROCESSING` 状态下

支持：
- 查看关联单据
- 跟踪仓库执行进度
- 查看是否已出库、是否已收货

## 9. 验收标准

1. 调拨单创建完成后，主状态显示为“已创建”，而不是直接进入执行态。
2. 页面可单独展示“执行单据创建状态”。
3. 当执行单据创建失败时，主状态保持“已创建”，执行单据创建状态显示“创建失败”。
4. 当执行单据部分成功时，主状态保持“已创建”，执行单据创建状态显示“部分成功”。
5. 当执行单据全部成功时，主状态可进入“执行中”。
6. 列表页新增“异常”Tab，并正确聚合失败、部分成功类单据。
7. 详情页可清晰区分调拨主状态与执行单据创建状态。

## 10. 本次不做

1. 不实现真实后端接口联动，仅先对前端状态与页面表达做统一。
2. 不引入复杂超时判定逻辑，异常 Tab 先按明确失败/部分成功数据聚合。
3. 不增加新的审批流或权限流。
