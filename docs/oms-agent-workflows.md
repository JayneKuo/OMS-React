# OMS Agent Workflows

本文档定义 OMS Agent 在复杂业务场景下的标准编排流程。当用户的问题不能由单个 skill 直接完成时，Agent 应参考本文件进行多 skill 编排。

> Skill 的详细定义、输入输出和错误码见 [Skill Registry](./oms-agent-skill-registry.md)。

---

## Workflow 路由规则

Agent 收到请求后，按以下规则选择 workflow：

```
用户请求
  ↓
1. 是否只需要查询单一数据？
   → 是：workflow 1（oms_query）
   → 否：继续
  ↓
2. 是否在问"为什么失败/异常"？
   → 是：workflow 2（exception_diagnosis）
   → 否：继续
  ↓
3. 是否只涉及分仓？（没提承运商/运费/时效）
   → 是：workflow 3（warehouse_allocation）
   → 否：继续
  ↓
4. 是否只涉及物流？（仓库已确定或不需要选仓）
   → 是：workflow 4（shipping_plan）
   → 否：继续
  ↓
5. 是否同时涉及分仓 + 物流？
   → 是：workflow 5（warehouse_plus_shipping）
   → 否：继续
  ↓
6. 以上都不匹配 → 单 skill 调用或引导用户澄清
```

### 路由升级规则

- 如果用户在 workflow 3 的结果上追问"那走哪个承运商"，自动升级到 workflow 5
- 如果用户在 workflow 4 的结果上追问"换个仓呢"，自动升级到 workflow 5
- 如果用户在 workflow 1 的结果上追问"为什么失败"，自动切换到 workflow 2

## Workflow 1：订单查询（oms_query）

### Trigger

- 查询订单状态/详情
- "SO12345 什么状态""这个订单到哪了""订单明细是什么"

### 输入要求

| 参数 | 必要性 | 说明 |
|------|--------|------|
| 订单号 / 订单标识 | 必须 | 如缺失则追问 |

### Steps

```
1. 识别订单号
   ↓
2. 调用 oms_query → 获取订单头、订单行、状态、履约状态、发货状态
   ↓ 成功                          ↓ 失败
3a. 输出关键结果              3b. 说明查询失败原因
    + 业务解释（如需要）           + 建议检查订单号是否正确
    + 页面链接                     + 提供订单管理页面链接
```

### 输出字段

| 字段 | 说明 |
|------|------|
| order_header | 订单头信息 |
| order_lines | 订单行明细 |
| order_status | 订单状态 |
| fulfillment_status | 履约状态 |
| shipment_status | 发货状态 |
| exception_summary | 异常摘要（如有） |

### 输出级别

`📋 查询结果`

---

## Workflow 2：异常诊断（exception_diagnosis）

### Trigger

- "这个订单为什么失败/异常"
- "为什么没有分仓/发货/生成标签/算出运费"
- "这个异常怎么处理"

### 输入要求

| 参数 | 必要性 | 说明 |
|------|--------|------|
| 订单号 | 必须 | 如缺失则追问 |
| 异常类型 | 可选 | 用户可能已说明，也可从数据中识别 |

### Steps

```
1. 调用 oms_query → 获取订单状态和上下文
   ↓ 成功                          ↓ 失败
   继续                           输出查询失败，建议检查订单号
   ↓
2. 调用 order_analysis → 获取失败节点和根因
   ↓ 成功                          ↓ 失败
   继续                           基于步骤 1 数据给出初步判断（标记为估算）
   ↓
3. 根据失败类型，选择性调用深度诊断：
   - 分仓相关 → warehouse_allocation（检查规则和候选仓）
   - 物流相关 → cartonization / shipping_rate / eta（检查依赖环节）
   - 地址相关 → 直接从订单数据分析
   ↓ 部分失败时
   说明哪些环节诊断成功、哪些失败，给出已知部分的结论
   ↓
4. 汇总输出
```

### 输出字段

| 字段 | 说明 |
|------|------|
| issue_type | 问题类型 |
| root_cause | 根因（确定时）/ 可能原因（不确定时） |
| impacted_step | 影响的业务环节 |
| recommendation | 修复建议（具体步骤） |
| retryable_flag | 是否可自动重试 |
| manual_action_needed | 是否需要人工处理 |

### 输出级别

- 根因明确时：`🔍 分析结论`
- 根因不确定时：`⚠️ 估算（仅供参考）` + 列出可能原因范围

## Workflow 3：分仓推荐（warehouse_allocation）

### Trigger

- "这单应该分哪个仓""推荐发货仓"
- "为什么分到这个仓""哪个仓最优"
- 注意：如果用户同时问了承运商/运费/时效，应路由到 workflow 5

### 输入要求

| 参数 | 必要性 | 说明 |
|------|--------|------|
| 订单号 或 SKU 列表 | 必须 | 至少提供一个 |
| 目的地 | 必须 | 从订单中获取或用户提供 |
| 约束条件 | 可选 | 用户指定的仓库偏好、时效要求等 |

### Steps

```
1. 调用 oms_query → 获取订单、商品、目的地
   ↓ 失败 → 追问订单号或 SKU + 目的地
   ↓
2. 调用库存/仓能力查询 → 获取各仓库存和能力
   ↓ 失败 → 说明无法获取库存数据，给出基于规则的初步建议（标记估算）
   ↓
3. 调用 warehouse_allocation → 生成候选仓和推荐仓
   ↓ 失败 → 基于步骤 2 的库存数据给出简单推荐（标记估算）
   ↓
4. 输出推荐仓 + 候选仓 + 推荐理由
```

### 输出字段

| 字段 | 说明 |
|------|------|
| recommended_warehouse | 推荐仓库 |
| candidate_warehouses | 候选仓库列表 |
| allocation_reason | 推荐理由（库存、距离、成本、规则等维度） |
| risks_and_constraints | 风险与限制 |

### 输出级别

`💡 推荐建议`

---

## Workflow 4：发货方案推荐（shipping_plan）

### Trigger

- "这单怎么发最合适""走哪个承运商""用什么服务"
- "这票货怎么装箱并发货""给我一个完整发货方案"
- 前提：仓库已确定（或用户不关心分仓）

### 输入要求

| 参数 | 必要性 | 说明 |
|------|--------|------|
| 订单号 或 SKU 列表 + 数量 | 必须 | 用于获取商品物理属性 |
| 发货仓 | 必须 | 从订单中获取或用户指定 |
| 目的地 | 必须 | 从订单中获取或用户提供 |
| 优先级偏好 | 可选 | 成本优先 / 时效优先 / 平衡（默认平衡） |

### Steps

```
1. 调用 oms_query → 获取订单、商品、目的地
   ↓ 失败 → 追问必要信息
   ↓
2. 获取 SKU 主数据与物理属性（重量、尺寸）
   ↓ 失败 → 说明缺少物理属性，后续运费为估算
   ↓
3. 调用 cartonization → 生成装箱方案
   ↓ 失败 → 跳过装箱，使用商品原始尺寸计算运费（标记估算）
   ↓
4. 调用 shipping_rate → 获取各承运商/服务报价
   ↓ 部分失败 → 说明哪些承运商查询失败，基于已有数据推荐
   ↓
5. 调用 eta → 获取各承运商/服务时效
   ↓ 失败 → 说明时效数据不可用，仅基于运费推荐
   ↓
6. 调用 cost → 计算综合成本
   ↓ 失败 → 基于运费数据给出简化推荐
   ↓
7. 综合排序，输出推荐方案 + 备选方案
```

### 输出字段

| 字段 | 说明 |
|------|------|
| package_plan | 装箱方案 |
| recommended_carrier | 推荐承运商 |
| recommended_service | 推荐服务 |
| estimated_shipping_cost | 预计运费 |
| estimated_eta | 预计时效 |
| total_cost | 综合成本 |
| recommendation_reason | 推荐理由 |
| alternative_options | 备选方案（至少 1 个） |

### 输出级别

- 所有 skill 成功：`💡 推荐建议`
- 部分 skill 失败：`💡 推荐建议` + `⚠️` 标注受影响的字段

## Workflow 5：分仓 + 物流综合推荐（warehouse_plus_shipping）

### Trigger

- "这单该分哪个仓、走哪个承运商、用什么服务"
- "给我完整推荐方案""从仓库到物流一起推荐"
- 从 workflow 3 或 4 升级而来

### 输入要求

| 参数 | 必要性 | 说明 |
|------|--------|------|
| 订单号 或 SKU 列表 + 数量 | 必须 | |
| 目的地 | 必须 | |
| 优先级偏好 | 可选 | 成本优先 / 时效优先 / 平衡 |
| 约束条件 | 可选 | 排除某仓库、指定承运商等 |

### Steps

```
1. 调用 oms_query → 订单、商品、目的地
   ↓
2. 调用库存/仓能力查询 → 各仓库存和能力
   ↓
3. 调用 warehouse_allocation → 候选仓列表（Top N）
   ↓
4. 对每个候选仓，并行执行：
   ├─ 调用 cartonization → 装箱方案
   ├─ 调用 shipping_rate → 运费
   ├─ 调用 eta → 时效
   └─ 调用 cost → 综合成本
   ↓
5. 生成组合方案矩阵：仓库 × 承运商 × 服务
   ↓
6. 按用户偏好（或默认平衡策略）排序
   ↓
7. 输出 Top 1 推荐 + Top 2-3 备选
```

### 错误处理

| 失败环节 | 影响 | 处理方式 |
|----------|------|----------|
| oms_query | 全流程无法执行 | 追问订单号或必要信息 |
| 库存查询 | 无法判断库存可用性 | 基于规则推荐，标记"未验证库存" |
| warehouse_allocation | 无法生成候选仓 | 基于库存数据简单排序，标记估算 |
| cartonization | 运费计算不精确 | 使用商品原始尺寸，标记"装箱未优化" |
| shipping_rate（部分） | 部分承运商缺失 | 说明哪些缺失，基于已有数据推荐 |
| shipping_rate（全部） | 无法推荐承运商 | 只输出分仓建议，说明运费查询失败 |
| eta | 无法比较时效 | 仅基于成本推荐，标记"时效未验证" |
| cost | 无法综合排序 | 基于运费 + 时效分别给出建议 |

### 输出字段

| 字段 | 说明 |
|------|------|
| recommended_warehouse | 推荐仓库 |
| recommended_carrier | 推荐承运商 |
| recommended_service | 推荐服务 |
| package_plan | 装箱方案 |
| estimated_shipping_cost | 预计运费 |
| estimated_eta | 预计时效 |
| total_cost | 综合成本 |
| recommendation_reason | 推荐理由（多维度） |
| alternative_options | 备选方案（含差异对比） |
| risks_and_constraints | 风险与限制 |

### 输出级别

`💡 推荐建议`（部分数据缺失时，受影响字段标注 `⚠️`）

---

## Workflow 使用原则

1. 能单 skill 解决的问题，不要走 workflow
2. 涉及多个业务维度的推荐问题，必须走 workflow
3. workflow 中每一步基于前一步结果，不得跳步
4. 某个关键步骤失败时，必须说明失败位置、原因和对最终结果的影响
5. 非关键步骤失败时，降级处理并标记受影响的输出字段
6. 数据不完整时可输出估算型结果，但必须标记为估算
7. 用户追问时可以从当前 workflow 升级到更复杂的 workflow，复用已有结果
8. 最终输出必须包含：结果 + 原因 + 风险/限制
