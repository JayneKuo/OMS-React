# 对象字段与信息架构模板

## 1. 目的

本文档定义每类核心业务对象应该展示哪些信息、这些信息应放在哪个区域，以及优先级顺序如何排列。

它是 UI 系统和业务 schema 之间的桥梁，解决的问题不是“页面怎么长”，而是“页面上应该出现什么”。

---

## 2. 信息优先级模型

每个对象都应拆成四个信息层级。

### Tier 1 —— 身份与当前状态
必须立即可见。
示例：
- 对象名称 / 编号
- 当前状态
- 关键金额 / 数量 / 负责人
- 关键时间戳

### Tier 2 —— 操作上下文
应出现在列表行、详情页头部或详情页首屏 section 中。
示例：
- 关联实体
- warehouse / supplier / customer
- channel / source / workflow stage

### Tier 3 —— 支撑细节
放在详情页主体或右侧 metadata 区。
示例：
- notes
- configuration flags
- internal reference IDs
- secondary timestamps

### Tier 4 —— 审计与长尾元数据
放在折叠区、次级 section、timeline 或 logs 中。
示例：
- raw metadata
- history events
- system sync traces
- internal debug info

---

## 3. 列表页与详情页放置规则

### 3.1 列表页
列表页只应展示：
- Tier 1 身份 / 状态
- 对当前决策最关键的 Tier 2 字段
- 必要时的 row-end action area

除非该页面本身是诊断控制台，否则不要把 Tier 3 / Tier 4 字段塞进列表中。

### 3.2 详情页
详情页应展示：
- Tier 1 在 header
- Tier 2 在首屏 section
- Tier 3 在 detail panels 或右侧 metadata blocks
- Tier 4 在 timeline / log / config / metadata sections

### 3.3 设置 / 配置页
设置类对象应优先展示：
- 模块名称
- scope
- 这项设置控制什么
- 高风险字段及其解释
- advanced fields 放在默认行为之后

---

## 4. Orders

### 4.1 列表页字段
默认必须列：
- Order number
- Status
- Customer / account
- Channel / source
- Order date
- Total amount
- Fulfillment / shipment state
- Exception indicator（如有）

可选字段：
- Warehouse
- Priority
- Assigned owner
- SLA due time

### 4.2 详情页头部
- Order number
- Current status
- Customer
- Order date
- Total amount
- Primary action cluster

### 4.3 详情页 sections
左侧 / 主区：
- line items
- fulfillment / shipment blocks
- payments / settlement（如相关）
- exception handling 或 timeline

右侧 / 次区：
- metadata
- tags
- addresses
- source identifiers
- sync / config blocks

### 4.4 展示规则
- status 是一级信息
- line items 永远高于 metadata
- shipping 与 payment 不能埋在普通字段堆里
- 如果有 exception state，应在 header 附近前置

---

## 5. Inventory / SKU

### 5.1 列表页字段
默认必须列：
- SKU / code
- Product / variant name
- Available quantity
- Reserved quantity
- Warehouse summary 或 primary warehouse
- Inventory status
- Updated at

可选字段：
- Inbound quantity
- Reorder threshold
- Supplier
- Category

### 5.2 详情页头部
- SKU
- Product name
- availability status
- total available quantity
- primary warehouse summary

### 5.3 详情页 sections
左侧 / 主区：
- warehouse quantities table
- stock movement / adjustments
- inbound / replenishment
- reservation breakdown

右侧 / 次区：
- identifiers
- vendor / category
- metadata
- planning thresholds

### 5.4 展示规则
- available stock 必须在视觉上压过 reserved 和 metadata
- 分仓明细应放在 summary 下方，而不是 summary 上方
- low stock / negative stock 等预警必须进入 Tier 1 或 Tier 2

---

## 6. Purchase Orders / Procurement Documents

### 6.1 列表页字段
默认必须列：
- PO number
- Supplier
- Status
- Created / ordered date
- Expected date
- Total amount
- Receiving progress

可选字段：
- Warehouse
- Buyer / owner
- Approval state

### 6.2 详情页头部
- PO number
- supplier
- status
- expected date
- total amount
- primary procurement actions

### 6.3 详情页 sections
左侧 / 主区：
- line items
- receiving progress
- shipment / ASN / supplier delivery info
- approvals / timeline

右侧 / 次区：
- payment terms
- supplier references
- attachments
- custom fields
- metadata

### 6.4 展示规则
- line items 永远是最高优先级内容
- supplier 与 status 属于 header 信息
- receiving state 不能藏进普通 metadata 中

---

## 7. Customers / Accounts

### 7.1 列表页字段
默认必须列：
- Customer name
- Email 或 primary contact
- Description / segment（如相关）
- Country / region
- Created date

可选字段：
- Lifetime value
- Active subscriptions
- Last order date
- Risk / verification status

### 7.2 详情页头部
- Customer name
- contact summary
- customer state
- top-level actions

### 7.3 详情页 sections
左侧 / 主区：
- orders / payments / subscriptions / activity
- 其他相关 financial 或 operational sections

右侧 / 次区：
- details
- metadata
- configuration

### 7.4 展示规则
- contact identity 属于 Tier 1
- 相关 activity sections 必须早于 raw metadata
- related-object 空状态 section 可以存在，但必须保持紧凑

---

## 8. Reports

### 8.1 报表 hub 入口字段
每个报表入口应至少展示：
- report name
- 它回答的 business question
- 必要时的生命周期标签（new / preview）

### 8.2 报表详情页字段
一个 report detail page 应至少包括：
- report title
- time period
- scope filters
- metric summary
- chart area
- detailed table 或 breakdown

### 8.3 展示规则
- summary metrics 在前
- filters 在 chart / table 主体之前
- 阅读顺序应稳定：summary -> trend -> breakdown -> raw rows

---

## 9. Approval / Exception Objects

### 9.1 队列列表字段
默认必须列：
- object ID / title
- current state
- owner / assignee
- age / SLA
- reason / category
- updated at

### 9.2 详情页 sections
左侧 / 主区：
- decision context
- core business data
- requested action / exception reason
- timeline

右侧 / 次区：
- metadata
- routing rules hit
- audit info

### 9.3 展示规则
- 审批原因 / 异常原因必须早于低价值 metadata
- due time / SLA 应进入 Tier 1 或 Tier 2

---

## 10. Settings / Configuration Modules

### 10.1 Hub card 字段
每个配置模块卡片应展示：
- module title
- 一句范围说明
- 如果配置未完成或不健康，可选状态提示

### 10.2 设置详情页分组顺序
推荐顺序：
1. default behavior
2. core options
3. integrations / dependencies
4. advanced settings
5. audit / last-modified info

### 10.3 展示规则
- 不要从 advanced settings 开始
- 高风险设置在控件前必须有解释上下文
- 罕见或危险选项必须使用渐进式披露

---

## 11. Developer / Diagnostic Surfaces

### 11.1 Event / log 列表字段
默认必须字段或 token：
- event / log ID
- timestamp
- status / result
- resource type
- event type / action type
- destination 或 subsystem（如相关）

### 11.2 诊断详情页 / drawer
- technical identifier
- current state / result
- related resource
- payload summary
- timeline 或 request / response details

### 11.3 展示规则
- 在 diagnostic view 中，IDs 与 timestamps 属于 Tier 1
- filters 必须快速暴露 date、status、type、resource
- debug info 可以更高密度，但层级仍必须清晰

---

## 12. AI 使用规则

当 AI 生成页面时，必须先从这些 schema 中选择字段。
除非用户明确要求，否则不得随意发明列，也不得省略预期的 Tier 1 字段。

提示词速记：
- 从 H-02-Stripe页面模板规范.md 选择 archetype
- 从 H-05-对象字段与信息架构模板.md 选择字段

- 用 H-04-AI执行约束规范.md 控制行为边界
